1. Executive Summary
NxtSft is an end-to-end real estate platform that connects home buyers, property owners, and sales
professionals through a unified, role-based portal suite. The platform eliminates brokerage fees for buyers,
provides verified property listings across 50+ Indian cities, and equips internal teams with purpose-built tools for
lead management, CRM, team supervision, customer support, and platform administration.
NxtSft is built on Next.js 15 with React 19 and TypeScript, delivering a fast, server-rendered, fully typed
application. The portal system is structured around seven distinct user roles, each with a dedicated interface
tailored to that role's specific workflows. All portals are accessible from a single domain and share a common
design language and authentication layer.

Key Platform Objectives
• Zero-brokerage property discovery for home buyers with RERA-verified listings.
• Token-based contact-unlock system to monetise buyer access to owner details.
• Full-funnel CRM for sales teams — from lead ingestion through site visit to close.
• Supervisor oversight with real-time team performance monitoring and escalation workflows.
• Centralised support ticketing with SLA enforcement and TAT reporting.
• Super Admin command centre for platform configuration, billing, AI tuning, and audit trails.

Key Metrics (Platform Goals)
• 10,000+ verified property listings across 12 major Indian cities.
• 50+ active cities covered via listing and search.
• 4.8 / 5.0 customer satisfaction rating target.
• 1,00,000+ registered customers.
• 100% RERA-verified listings.
• ¹0 brokerage for home buyers.

2. Platform Overview & Architecture
2.1 Technical Stack
• Framework: Next.js 15 with App Router (React Server Components + Client Components)
• Language: TypeScript 5.8 — fully typed throughout
• Styling: Tailwind CSS 4 with tw-animate-css for motion utilities
• UI Library: Radix UI primitives (Dialog, Tabs, Progress, Tooltip, Separator, Slot)
• Icons: Lucide React icon library
• Toasts: Sonner notification library
• Auth: Custom localStorage-based session system (demo); production-ready role model
• Data: Static JSON fixtures (demo); pluggable to REST / Supabase in production
• Build: Vercel-compatible deployment; standard npm dev / build / start scripts

2.2 Application Structure
• src/app/
— Next.js App Router pages and layouts
• src/components/ — Shared UI components (portal shells, site header/footer, etc.)
• src/lib/
— Auth context, utilities, hooks
• src/data/
— Static data fixtures for properties, leads, teams, plans, etc.

2.3 Portal Architecture
The application uses a single shared shell component (PortalShell) that renders the sidebar navigation, top
header, role badge, and brand identity. Each portal passes its navigation items and active tab state to this shell.
Role-based routing ensures users are automatically redirected to the correct portal after authentication.
• PortalShell: reusable layout wrapper used by all six staff/user portals
• useActiveHash: React hook for route-hash–driven active tab management
• StatCard: shared KPI metric card with icon, value, label, and optional delta
• Section: content block wrapper with title and optional action button
• Badge: status pill supporting tones — hot, warm, cold, new, success, default
• ReportsDashboard: shared reports component with filtering and export, embedded in multiple portals

2.4 Core Domain Concepts
• Property: Real estate listing with full specs, images, RERA number, pricing, and owner contact
• Lead: Prospective buyer captured through the platform, assigned to a sales rep for follow-up
• Credit / Token: Platform currency used by buyers to unlock owner contact details; purchased via
subscription plans
• Site Visit: Scheduled property inspection by a buyer, coordinated through a sales rep
• Subscription Plan: Tiered plan purchased by buyers or owners to access platform features
• RERA: Real Estate Regulatory Authority registration number validating a listing or owner
• Pipeline: CRM funnel stages tracking a lead from first contact through to deal closure

3. User Roles & Permission Hierarchy
NxtSft defines seven distinct user roles. Each role maps to a dedicated portal URL and a specific set of
capabilities. Role assignment is determined at login and cannot be changed by the user; only an admin or super
admin can reassign roles.

Role 1 — Super Admin (super-admin)
• Portal URL: /sa-portal
• Access Level: Full system access. Manages platform config, all users, billing, AI models, audit logs, and
role permissions. No restrictions.
• Demo Email: super-admin@nxtsft.com
Role 2 — Admin (admin)
• Portal URL: /admin-portal
• Access Level: Operations manager. Manages team, listings, leads, CRM pipeline, subscriptions, plans,
commissions, and marketing. Cannot access super-admin controls.
• Demo Email: admin@nxtsft.com
Role 3 — Supervisor (supervisor)
• Portal URL: /supervisor-portal
• Access Level: Team lead. Monitors team performance, reassigns leads, reviews escalations, manages
visit calendar. Cannot edit plans or billing.
• Demo Email: supervisor@nxtsft.com
Role 4 — Sales Rep (sales)
• Portal URL: /sales-portal
• Access Level: Field agent. Manages own leads, logs calls and visits, tracks commission. Scoped to their
own lead pool.
• Demo Email: priya@nxtsft.com
Role 5 — Support Admin (support-admin)
• Portal URL: /support-portal
• Access Level: Customer support agent. Manages support ticket queue, escalations, TAT reporting, and
knowledge base.
• Demo Email: support-admin@nxtsft.com
Role 6 — Home Buyer (user)
• Portal URL: /user-portal
• Access Level: End consumer. Searches and saves properties, uses credits to unlock contacts, schedules
site visits, manages KYC.
• Demo Email: rohan@example.com
Role 7 — Customer (customer)
• Portal URL: /user-portal
• Access Level: Premium concierge buyer. Same as Home Buyer but designated as a managed/priority
customer with Concierge portal label.
• Demo Email: ananya@example.com
Staff roles (super-admin, admin, supervisor, sales, support-admin) log in via /admin-login. Consumer roles (user,
customer) log in via /login. Registration at /register creates a user role session and grants 1 free welcome credit.

13. Authentication, Session & Security
13.1 Auth Architecture
NxtSft uses a React Context-based auth layer (AuthProvider / useAuth hook) backed by localStorage for the
demo implementation. In production this would be backed by a secure server-side session or JWT system.
• Session storage key: "nxtsft.session" in localStorage
• Credits storage key: "nxtsft.credits" in localStorage
• User registry key: "nxtsft.users" array in localStorage (demo registration)

13.2 Auth Functions
• signIn(role): Creates a session from ROLE_META for the given role. Grants 3 free credits to user/
customer roles if balance is zero.
• signOut(): Removes the session from localStorage. Credits are intentionally preserved across sign-out.
• register(name, email, phone): Creates a user-role session, persists to user registry, grants 1 welcome
credit if balance is zero.
• updateProfile(name, phone): Mutates the current session name, phone, and derived initials in
localStorage.
• addCredits(n): Increments the stored credit balance by n.
• useCredit(): Decrements credit balance by 1; returns false if balance is zero (contact unlock gate).

13.3 Role-Based Routing
• Staff roles (super-admin, admin, supervisor, sales, support-admin) !’ /admin-login
• Consumer roles (user, customer) !’ /login and /register
• On successful sign-in, client redirects to ROLE_META[role].portal
• Unauthenticated access to portal URLs redirects to the appropriate login page

13.4 Security Features (User Portal)
• Password change form: current password verification, new password, confirm password
• 2FA via OTP: toggle to enable one-time password for login
• Active sessions list: view all active login sessions, option to terminate specific sessions

13.5 Security Features (Super Admin)
• Failed login tracking with IP address logging
• IP whitelist / blacklist management
• Force-terminate any user session globally
• 2FA enforcement policy per role
• Password policy configuration: length, complexity, expiry
• Immutable audit trail of all admin actions

13.6 RERA Compliance
• Every listed property must provide a valid RERA registration number
• Owner/agent RERA badge is displayed on property listings and agent profiles
• RERA number is displayed prominently on property detail pages
• Platform validates RERA format during listing submission
